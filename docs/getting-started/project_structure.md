# Project structure

**What you'll learn:** Where files live in a typical reflex-django project and which paths are yours to edit versus generated.

**When you need this:**

- You are starting a new project and want a sensible folder layout before writing pages.
- You are adding Reflex to an existing Django repo and need to know what stays the same.

---

reflex-django does not impose a special layout. Use the same structure you would in any Django project. The layout below is what most teams find easiest because pages and models stay in the same app folder.

---

## Recommended layout

```text
myproject/
├── manage.py
├── pyproject.toml
├── db.sqlite3
│
├── config/                       # Django project package
│   ├── settings.py               # INSTALLED_APPS, MIDDLEWARE, REFLEX_DJANGO_*
│   ├── urls.py                   # Django routes + import page modules
│   ├── asgi.py                   # get_asgi_application()  -  plain Django ASGI
│   └── wsgi.py
│
├── shop/                         # a Django app
│   ├── models.py                 # ORM models
│   ├── admin.py
│   ├── views.py                  # @page functions and AppState subclasses
│   ├── serializers.py            # optional ReflexDjangoModelSerializer
│   └── migrations/
│
├── blog/                         # another app with its own views.py
│   ├── models.py
│   └── views.py
│
└── .web/                         # generated by Reflex (gitignore)
```

Three things to notice:

1. **One `manage.py`** at the repo root, same as any Django project.
2. **Pages and models share an app folder.** `shop/models.py` and `shop/views.py` sit next to each other.
3. **No `rxconfig.py`. No `shop/shop.py`.** Config lives in `settings.py`. The Reflex app singleton is `from reflex_django import app` (backed by `reflex_django.runtime.reflex_app`).

For the mental model behind those three knobs, see [The three knobs](../overview/concepts.md).

---

## What goes where

| Location | What lives there |
|:---|:---|
| `config/settings.py` | Django apps, middleware, database, `REFLEX_DJANGO_*` overrides |
| `config/urls.py` | Django routes; `import shop.views` to register pages at import time |
| `config/asgi.py` | Plain Django ASGI: `get_asgi_application()` |
| `{app}/models.py` | Django ORM models (unchanged) |
| `{app}/views.py` | `@page`-decorated Reflex pages and `AppState` subclasses |
| `{app}/serializers.py` | Optional `ReflexDjangoModelSerializer` classes |
| `{app}/admin.py` | Django admin registrations (unchanged) |
| `{app}/migrations/` | Django migrations (unchanged) |
| **Not present** | `{app}/{app}.py`, the shared Reflex app is auto-loaded |
| **Not present** | `rxconfig.py`, config lives in `settings.py` |

## Generated and ignored paths

These directories are created by Reflex or `manage.py run_reflex`. Do not edit them. Add them to `.gitignore`:

```gitignore
.web/
.reflex/
staticfiles/
```

| Path | What's in it |
|:---|:---|
| `.web/` | Reflex frontend project (compiled React source) |
| `.web/build/client/` | Compiled SPA bundle (SSR layout) |
| `.web/_static/` | Compiled SPA bundle (legacy layout) |
| `.reflex/` | Reflex per-user cache |
| `STATIC_ROOT/_reflex/` | Staged SPA assets served in production |

## How pages get registered

**Recommended (v1):** import page modules in `urls.py` so `@page` runs when Python imports the module:

```python
import shop.views  # noqa: F401
```

That import runs at **import/compile time**, not lazily on the first HTTP request. Before each Reflex compile, reflex-django also calls `prepare_pages_for_compile()` to import page packages and sync routes.

**Deprecated:** with `REFLEX_DJANGO_AUTO_DISCOVER_PAGES=True` (still the default), reflex-django scans `INSTALLED_APPS` and imports every `{app}.views` during compile preparation. You may see a deprecation warning. Prefer explicit imports before auto-discover is removed in a future major release.

Auto-discover skips `django.*` apps and `reflex_django` itself. If `{app}/views.py` does not exist, that app is silently skipped. Import errors surface in the console.

## The `app_name` setting

Put `app_name` in `REFLEX_DJANGO_RX_CONFIG`. It is Reflex's **compile label** (project id), not a rule that all pages must live in `{app_name}/views.py`.

- Appears in compiled artifacts and `DECORATED_PAGES` grouping
- If auto-discover is off, only `{app_name}.views` is imported unless you set `REFLEX_DJANGO_PAGE_PACKAGES`
- Can differ from where pages actually live (for example `app_name: "core"` with pages in `modules.ai.studio.views`)

If you omit it, reflex-django uses the folder name containing `manage.py` (hyphens become underscores). See [The three knobs](../overview/concepts.md#what-is-app_name).

## Alternative layout: pages in a separate package

Some teams keep UI code in its own package:

```text
myproject/
├── shop/                      # models, admin, business logic
│   ├── models.py
│   └── admin.py
└── frontend/                  # all Reflex pages
    ├── pages/
    │   ├── home.py
    │   ├── catalog.py
    │   └── account.py
    └── views.py               # re-exports each page
```

Two ways to wire it:

**Option A**, add `frontend` to `INSTALLED_APPS` and import `frontend.views` in `urls.py`.

**Option B**, list modules explicitly:

```python
REFLEX_DJANGO_PAGE_PACKAGES = [
    "frontend.pages.home",
    "frontend.pages.catalog",
    "frontend.pages.account",
]
```

A non-empty `REFLEX_DJANGO_PAGE_PACKAGES` disables auto-discovery.

## Static files

| Environment | How it works |
|:---|:---|
| **Development** | `manage.py run_reflex` builds the SPA into `STATIC_ROOT/_reflex/`. Django's ASGI static handler serves admin assets and the SPA. |
| **Production** | Run `python manage.py export_reflex --frontend-only --no-zip --stage-to-static-root` in CI, then `collectstatic`, then serve `/static/` with Nginx or your ASGI process. |

See [Deployment](../operations/deployment.md) for the full production setup.

!!! tip "Splitting a large views.py"
    Turn `views.py` into a package (`views/__init__.py` re-exporting submodules). Explicit `import shop.views` still loads everything as long as `__init__.py` imports the page modules.

You saw the recommended Django layout with Reflex pages in `{app}/views.py`, the four files you wire once (settings, urls, asgi, views), and which folders Reflex generates.
